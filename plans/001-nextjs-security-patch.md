## Why this matters

The repo pins `next@14.2.18`. `npm audit` reports **GHSA-f82v-jwr5-mffw — "Authorization Bypass in Next.js Middleware" (critical, CVSS 9.1)** against this version, plus several moderate advisories. This app's _entire_ auth wall is implemented in middleware (`middleware.ts` → `lib/supabase/middleware.ts` redirects unauthenticated users to `/login`), which is exactly the surface this CVE bypasses: a crafted `x-middleware-subrequest` header makes Next skip middleware entirely, exposing every signed-in page and server-rendered data. The fix was backported to the 14.2.x line (first fixed in 14.2.25); the newest 14.x release is **14.2.35** (the `next-14` npm dist-tag as of 2026-06-10). This plan bumps within 14.2.x — a patch-level change with negligible breakage risk. A full Next 15/16 migration is intentionally **not** part of this plan.

## Steps

### Step 1: Bump next and eslint-config-next to 14.2.35

In `package.json`, change:

- `"next": "14.2.18"` → `"next": "14.2.35"`
- `"eslint-config-next": "14.2.18"` → `"eslint-config-next": "14.2.35"`

Then run `npm install`.

**Verify**: `npm ls next` → shows `next@14.2.35`, exit 0.

### Step 2: Confirm the critical advisory is gone

**Verify**: `npm audit --omit=dev 2>&1 | tail -5` → the summary line no longer contains `1 critical`. GHSA-f82v-jwr5-mffw must not appear in the output. (Moderate advisories fixed only in Next 15/16 may remain — that is expected; record the remaining count in your report.)

### Step 3: Fix the transitive `ws` advisory

Run `npm audit fix` (NOT `--force` — `--force` would install Next 16, a breaking change).

**Verify**: `npm audit --omit=dev 2>&1 | tail -5` → `ws` no longer listed. `git diff package.json` → no unexpected version changes beyond `next`, `eslint-config-next`, and lockfile-only transitive bumps.

### Step 4: Full verification suite

**Verify**, in order:

1. `npm run lint` → "No ESLint warnings or errors"
2. `npm run typecheck` → exit 0
3. `npm test` → 30 tests pass
4. `npm run build` → exit 0 (this is the real compatibility gate for a framework bump)

### Step 5: Smoke-test the auth wall

Start the dev server (`npm run dev`) and, with no auth cookie:

**Verify**: `curl -s -o /dev/null -w "%{http_code} %{redirect_url}" http://localhost:3000/feed` → `307` (or `302`) with redirect URL containing `/login?next=%2Ffeed`. Then stop the dev server.
